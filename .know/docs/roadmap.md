# know 产品路线图

<!-- 核心问题: 产品到哪了、下一步往哪走？ -->

## 1. 产品愿景

| | |
|---|---|
| **解决什么问题** | AI agent 跨会话丢失隐性知识和设计讨论结果，重复犯错 |
| **给谁用** | 使用 Claude Code 等 AI 编程工具的开发者 |
| **核心差异** | 知识与代码同源同版本、分层控制 token 成本、自动衰减保持新鲜 |

## 2. 版本规划

### v1 — 两条管线闭环

| 维度 | 标准 |
|------|------|
| **功能** | learn（知识沉淀）+ write（文档输出）两条管线闭环 |
| **质量** | 真实项目跑通全流程，无阻塞性 bug |
| **用户** | 作者本人日常使用，验证核心假设 |

#### 里程碑

| # | 里程碑 | 验证点 | 进度 | 需求 |
|---|--------|--------|------|------|
| M1 | **能存** — learn 管线端到端跑通 | 一条知识从对话检测 → 确认 → 写入 `.know/`，无手动干预 | 1/1 | [learn](.know/docs/requirements/know-learn/prd.md) |
| M2 | **能写** — write 管线端到端跑通 | 一次对话讨论 → `/know write` → 生成完整文档 + 索引更新 | 1/1 | [write](.know/docs/requirements/know-write/prd.md) |
| M3 | **能防** — 沉淀的知识实际阻止了一次重复犯错 | agent 在新会话中命中已有知识，避免了已知错误 | 1/1 | — |
| M4 | **能用** — 真实项目连续使用 ≥1 周 | 日常开发无阻塞，知识条目 ≥10 条，命中率 >0 | 1/1 | — |

### v2 — 可观测性

| 维度 | 标准 |
|------|------|
| **功能** | learn/write/recall 各有核心指标，可通过命令查看 |
| **质量** | 指标数据来自真实使用，非模拟数据 |
| **用户** | 看到指标后能判断"哪里需要优化"并采取行动 |

#### 里程碑

| # | 里程碑 | 验证点 | 进度 | 需求 |
|---|--------|--------|------|------|
| M1 | **能量** — 核心指标定义 + 数据收集 | `know-ctl metrics` 输出 learn/recall/write 各 2-3 个指标，数据来自真实使用 | 1/1 | [metrics](.know/docs/requirements/know-metrics/prd.md) |
| M2 | **能看** — 知识生命周期可追踪 | 一条知识从存入 → 命中 → 衰减/删除的完整轨迹可查询 | 1/1 | [lifecycle](.know/docs/requirements/know-lifecycle/prd.md) |
| M3 | **能调** — 基于数据做出一次优化决策 | 看到指标后发现问题，调整规则，指标改善 | 1/1 | [optimize](.know/docs/requirements/know-optimize/prd.md) |

### v3 — 可靠性

| 维度 | 标准 |
|------|------|
| **功能** | know-ctl 有自测、文档有一致性检查、write 流程规则完整无遗漏 |
| **质量** | self-test 全部通过、check 输出 0 个误报 |
| **用户** | 日常使用中不再遇到路径错误、版本判定错误、文档不同步 |

#### 里程碑

| # | 里程碑 | 验证点 | 进度 | 需求 |
|---|--------|--------|------|------|
| M1 | **能测** — know-ctl 自动化验证 | `know-ctl self-test` 覆盖核心命令（append/query/hit/decay/metrics/history），全部通过 | 1/1 | [selftest](.know/docs/requirements/know-selftest/prd.md) |
| M2 | **能查** — 文档一致性检查 | `know-ctl check` 检测模版与项目文档的结构偏差，输出具体不一致项 | 1/1 | [check](.know/docs/requirements/know-check/prd.md) |
| M3 | **能守** — 流程规则完整性 | write 流程的关键判定点（版本、类型、父子关系）有明确规则且无遗漏 | 1/1 | [flowrules](.know/docs/requirements/know-flowrules/prd.md) |

## 3. 当前版本

v3 — 可靠性（已完成）

### 包含

| 能力 | 解决什么问题 |
|------|-------------|
| **自测** | know-ctl 改动后无验证手段 → self-test 命令一键验证核心功能 |
| **一致性检查** | 模版改了但项目文档没跟上 → check 命令检测结构偏差 |
| **流程规则** | write 判定规则不完整导致写错文件 → 关键判定点有明确规则 |

### 排除

- 自动修复不一致（只检测，不自动改）
- 跨项目一致性检查
- CI/CD 集成
