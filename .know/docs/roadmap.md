# know 产品路线图

## 1. 产品愿景

### 产品核心

- **定位**: 面向 AI 编程助手的项目知识编译器——将隐性知识结构化持久化，让 agent 修改代码前自动获取相关上下文
- **动机**: AI agent 跨会话协作时无法保留设计决策、踩坑记录、模块间隐含约束，导致重复犯错和无效探索；现有方案（CLAUDE.md 手动维护）随条目增长 token 成本不可控且无分层管理
- **长远愿景**: AI agent 在项目中像有经验的团队成员一样工作——了解历史、尊重约束、不重蹈覆辙，项目知识像代码一样可版本化、可检索、可自动衰减

### 价值体系

| 层级 | 价值 | 衡量指标 |
|------|------|---------|
| **即时价值** | 消除跨会话上下文重建成本 | recall 防御次数（当前实测: 2 次）、recall 查询耗时（实测: <1s） |
| **累积价值** | 项目知识资产持续沉淀 | 知识条目总数（当前实测: 54 条）、recall 命中率（当前实测: 1%，目标: >30%，待优化） |
| **战略价值** | AI 协作从无状态对话升级为有记忆的持续协作 | 新会话首次有效协作时间（目标值，待验证: <2min）、跨会话知识复用率（无数据，尚无测量手段） |

### 核心问题

| 问题 | 发生频率 | 单次成本 | 影响面 | 现有应对 |
|------|---------|---------|--------|---------|
| AI 在新会话中重复犯已知错误 | 每次新会话（估算: 日均 5-20 次，基于作者使用频率） | 估算 10-30 分钟重新探索 + 引入回归 bug 风险（基于作者体感，未精确计时） | 所有使用 AI 助手维护中大型项目的开发者 | 手动维护 CLAUDE.md，条目膨胀后 token 成本不可控且无法分层检索 |
| 设计讨论结果无法跨会话传递 | 每次涉及架构/方案决策的会话 | 估算返工 1-4 小时（基于作者体感） | 多会话迭代的中大型项目 | 依赖 git commit message 和代码注释，覆盖面不足，缺乏结构化检索能力 |
| 知识老化无人清理 | 随项目演进持续累积 | 过时知识误导 agent，比没有知识更危险（定性判断，无量化数据） | 长期维护的项目（>1 月） | 无自动化衰减机制，全靠人工定期清理，实际几乎无人执行 |

### 目标用户

| 角色 | 典型场景 | Before | After | 预估提效 |
|------|---------|--------|-------|---------|
| 使用 Claude Code 的独立开发者 | 跨会话维护中大型项目，需 AI 记住项目约束和历史决策 | 每次新会话花数分钟重建上下文，关键约束仍会遗漏，已知 bug 重复出现（估算，未精确计时） | agent 自动获取相关知识，已知错误可被拦截（已验证 2 次防御成功） | 待验证（预期: 单次会话节省数分钟，回归 bug 减少） |
| AI 编程工具的 skill/plugin 开发者 | 为 AI agent 构建项目级上下文层，减少 agent 探索错误 | 需自建知识存储方案，维护成本高，格式不统一 | 开箱即用的知识基础设施，标准化存储 + 检索 + 衰减 | 待验证（预期: 省去自建基础设施的开发量） |

### 竞品对比

| 方案 | 定位 | 目标用户 | 核心功能 | 优势 | 局限 |
|------|------|---------|---------|------|------|
| **know** | AI agent 项目知识编译器 | Claude Code 用户、skill 开发者 | 自动提取、分层存储、按 scope 召回、自动衰减、结构化文档生成 | 知识自动提取无需手动维护，分层控制 token 成本，衰减保持新鲜 | 目前仅支持 Claude Code 生态，单项目知识库不跨项目 |
| CLAUDE.md 手动维护 | 项目级 AI 提示文件 | 所有 Claude Code 用户 | 手动编写项目规则和上下文 | 零依赖，开箱即用，完全可控 | 条目膨胀后 token 成本线性增长，无分层无检索，无自动清理 |
| Cursor Rules | IDE 级 AI 行为配置 | Cursor IDE 用户 | 项目规则配置、行为约束 | Cursor 深度集成，IDE 内无缝体验 | 仅适用于 Cursor，无知识提取能力，无生命周期管理 |
| git commit + 代码注释 | 代码级历史记录 | 所有开发者 | 变更历史、内联注释 | 自然产生无额外成本，与代码共存 | 分散在历史中无法结构化检索，无法区分知识重要程度，无法主动注入 |

## 2. 版本规划

### 汇总

| 版本 | 核心方向 | 核心指标变化 | 状态 | 周期 | 里程碑 |
|------|---------|------------|------|------|--------|
| v1 — 闭环验证 | 验证端到端闭环可行性 | ↑ 知识条目从 0 开始积累，recall 防御从 0 开始生效；↓ 无基线对比数据 | 已发布 | 2025.04.08 - 2025.04.11 | M1-M4 |
| v2 — 可观测性 | 让知识库运行状态可见可调 | ↑ 指标覆盖 0→3 管线（learn/recall/write）；↓ 无量化对比（v1 无指标体系） | 已发布 | 2025.04.12 - 2025.04.13 | M5-M7 |
| v3 — 可靠性 | 消除日常使用中的可靠性障碍 | ↑ 测试覆盖 0→6 命令，一致性检测 0→3 类偏差；↓ 路径解析 bug 已消除 | 已发布 | 2025.04.14 - 2025.04.16 | M8-M10 |

### 版本详情

#### v1 — 闭环验证

- **战略意图**: 验证核心假设——AI agent 能否通过持久化知识减少跨会话犯错。产品愿景成立的前提是 learn→recall 闭环跑通
- **投入产出**: 投入 4 天全职开发 → 预期建立端到端知识存取能力（收益待后续版本量化验证）
- **优先级依据**: 核心闭环未验证前后续版本无意义，无外部依赖，可立即启动
- **风险与依赖**: 依赖 Claude Code plugin 机制稳定性；风险：JSONL 在大量条目时性能未验证，AI 信号检测准确率未知
- **成功指标**: learn 管线端到端跑通，write 管线端到端跑通，至少 1 次真实 recall 防御成功，连续使用 ≥1 周无阻塞
- **核心价值**:
  1. 首次可以从对话自动提取知识并持久化到项目
  2. 首次可以从对话讨论生成结构化文档
  3. 知识可以在新会话中自动召回并阻止已知错误
- **用户覆盖**: 作者 dogfood
- **核心指标**（基线 → v1）:

| 指标 | 基线 | v1 | 变化 | 来源 |
|------|------|-----|------|------|
| 知识条目总数 | 0 | 无逐版本拆分数据（当前累计 54） | ↑ | 无逐版本快照，know-ctl stats 仅有当前值 |
| recall 防御次数 | 0 | 无逐版本拆分数据（当前累计 2） | ↑ | know-ctl metrics |
| learn 管线可用 | 否 | 是 | 新增 | 功能验证 |
| write 管线可用 | 否 | 是 | 新增 | 功能验证 |
| 文档类型支持数 | 0 | 10 | ↑ 10 | 模版文件计数 |
| 阻塞性 bug | N/A | 0（v1 周期内） | — | 使用记录 |

#### v2 — 可观测性

- **战略意图**: v1 闭环跑通后无法判断知识库健康度——条目是否有用、衰减是否合理、哪个管线需要优化。需要数据驱动决策能力
- **投入产出**: 投入 2 天开发 → 预期建立持续优化能力，避免知识库质量随时间退化
- **优先级依据**: v1 已验证闭环可行，但日常使用中发现无法判断"存的知识有没有用"，优化无抓手
- **风险与依赖**: 依赖 v1 产生的真实使用数据；风险：指标定义可能不准确，需要迭代调整
- **成功指标**: metrics 命令输出 learn/recall/write 各 2-3 个指标；基于数据做出 ≥1 次有效优化决策
- **核心价值**:
  1. 核心运营指标（learn/recall/write）可通过命令查看
  2. 知识全生命周期（存入→命中→衰减→删除）可追踪
  3. 基于数据发现问题→调整规则→验证改善的闭环
- **用户覆盖**: 作者 dogfood
- **核心指标**（v1 → v2）:

| 指标 | v1 | v2 | 变化 | 来源 |
|------|-----|-----|------|------|
| 指标覆盖管线数 | 0 | 3（learn/recall/write） | ↑ 3 | know-ctl metrics 功能验证 |
| 知识生命周期可追踪 | 否 | 是（history 命令） | 新增 | 功能验证 |
| 数据驱动优化次数 | 0 | ≥1（具体优化内容待查证） | ↑ | 使用记录，待确认具体优化案例 |
| recall 命中率 | 无数据（v1 无指标） | 1%（实测 1/54） | 新增基线 | know-ctl metrics |

#### v3 — 可靠性

- **战略意图**: v1/v2 日常使用中暴露可靠性问题——路径解析 bug 调试困难、模版改了文档没跟上、write 判定规则有遗漏。需要消除这些阻碍持续使用的障碍
- **投入产出**: 投入 3 天开发 → 预期消除日常使用中的 3 类可靠性问题，降低维护心智负担
- **优先级依据**: v2 的数据能力已就绪，但可靠性问题导致维护成本高于预期，不解决会影响推广
- **风险与依赖**: 依赖对 v1/v2 使用中暴露问题的完整梳理；风险：check 命令的模版解析可能误判
- **成功指标**: self-test 全部通过，check 输出 0 个误报，write 关键判定点无规则遗漏
- **核心价值**:
  1. know-ctl 有自动化回归测试，改动后一键验证
  2. 模版与文档一致性可机器检测
  3. write 流程关键判定点有明确规则，不会因规则遗漏写错文件
- **用户覆盖**: 作者 dogfood
- **核心指标**（v2 → v3）:

| 指标 | v2 | v3 | 变化 | 来源 |
|------|-----|-----|------|------|
| self-test 覆盖命令数 | 0 | 6 | ↑ 6 | know-ctl self-test 实测 |
| self-test 通过率 | N/A | 100% | 新增 | know-ctl self-test 连续运行验证 |
| 文档一致性检测偏差类型 | 0 | 3（缺失/多余/层级） | ↑ 3 | know-ctl check 功能验证 |
| write 规则覆盖判定点 | 部分 | 全部（版本/类型/父子/模式） | ↑ | write.md 规则审查 |
| 知识条目总数 | 无逐版本数据 | 54（当前累计） | — | know-ctl stats |
| recall 命中率 | 1% | 1%（未变化） | — | know-ctl metrics，命中率低待 v4 优化 |

#### v4 — recall 有效性

- **战略意图**: v1-v3 建立了知识存取闭环，但 recall 命中率仅 1%（1/73），73 条知识中 72 条从未被使用。存的知识不被用等于没存。v4 解决"存了但用不上"的核心瓶颈
- **投入产出**: 预期 1-2 周开发 → 预期命中率从 1% 提升到 >30%，recall 防御次数从 2 提升到 >10
- **优先级依据**: recall 是 know 的核心价值（防错），命中率 1% 意味着核心价值未被兑现。不解决则知识库越大越浪费 token。推广前必须先证明有效性
- **风险与依赖**: 依赖 v3 的 metrics/review 能力提供数据支撑；风险：命中率低可能是知识质量问题（存了无用的）或匹配机制问题（scope 推断过窄），需先诊断再行动
- **成功指标**: recall 命中率 >30%，scope 覆盖率 >50%，防御次数 >10
- **核心价值**:
  1. 知识库从"存了就忘"变为"存了就用"
  2. recall 的防错价值可被数据证明
  3. 用户可信赖知识库——存的东西真的会在关键时刻出现
- **用户覆盖**: 作者 dogfood
- **核心指标**（v3 → v4）:

| 指标 | v3 | v4 目标 | 变化 | 来源 |
|------|-----|---------|------|------|
| recall 命中率 | 1%（1/73） | >30% | 目标值，待验证 | know-ctl metrics |
| recall 覆盖率 | 9%（3/32 scope） | >50% | 目标值，待验证 | know-ctl metrics |
| recall 防御次数 | 2 | >10 | 目标值，待验证 | know-ctl metrics |
| 知识条目总数 | 73 | 预期减少（review 清理后） | — | know-ctl stats |
| 有效条目比例 | 无数据 | >60%（命中过或 <30 天的条目占比） | 目标值，待验证 | 待新增指标 |

#### v5 — recall 可观测

- **战略意图**: v4 让知识被用起来，但 recall 的运行过程是黑盒——触发了多少次、匹配了什么、空查了多少次都不可见。没有过程数据就无法持续优化 recall 规则。v5 补齐 recall 运行数据采集
- **投入产出**: 预期 1-2 天开发 → 预期建立 recall 全链路可观测能力，为后续优化提供数据基础
- **优先级依据**: v4 M13 需要 2 周实际使用数据验证 recall 价值，但当前无法区分"recall 没触发"和"触发了但没命中"——v5 的数据采集直接支撑 M13 的价值量化
- **风险与依赖**: 依赖 v4 完成（recall 有效后数据才有意义）；风险：事件写入增加 I/O 开销（缓解：仅追加 1 行 JSONL，<1ms）
- **成功指标**: recall_query 事件可记录，metrics 可展示 recall 触发/命中/空查分布
- **核心价值**:
  1. recall 从黑盒变为白盒——每次触发都有记录
  2. 可区分"没触发"和"触发了没命中"，精准定位优化方向
  3. scope 盲区可被数据发现（哪些 scope 从未被查询）
- **用户覆盖**: 作者 dogfood
- **核心指标**（v4 → v5）:

| 指标 | v4 | v5 目标 | 变化 | 来源 |
|------|-----|---------|------|------|
| recall 触发次数 | 无数据（未采集） | 可统计 | 新增 | events.jsonl recall_query 事件 |
| recall 空查率 | 无数据（未采集） | 可统计 | 新增 | recall_query where matched=0 |
| scope 查询覆盖 | 无数据（未采集） | 可统计 | 新增 | unique(recall_query.scope) |
| metrics recall 面板 | 防御次数+覆盖率 | 新增触发/命中/空查分布 | ↑ | know-ctl metrics 扩展 |
