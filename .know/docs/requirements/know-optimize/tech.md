# know optimize 技术方案

## 1. 背景

### 技术约束

- 命令边界: 建议逻辑嵌入 cmd_metrics 末尾输出，不新增独立命令
- 数据来源: 只使用 M1 metrics 已有指标 + M2 lifecycle 已有历史，不新增数据收集点
- 阈值策略: PRD 排除自定义阈值，所有阈值硬编码

### 前置依赖

- M1 cmd_metrics 指标输出（6 个指标） — 已完成
- M2 lifecycle events 历史记录 — 已完成
- workflows/review.md 排序规则 — 已完成

## 2. 方案

### 文件/模块结构

| 文件/模块 | 职责 |
|-----------|------|
| scripts/know-ctl.sh `cmd_metrics` 末尾 | 指标输出后追加建议生成逻辑，逐条检查阈值并输出行动建议 |
| workflows/review.md Step 2 | 排序规则改为生命阶段优先（濒危 → 沉默 → 新建 → 活跃） |

### 核心流程

1. `cmd_metrics` → 输出 6 个指标后逐条检查阈值（命中率 < 50%、衰减率 > 30%、防御次数 = 0、覆盖率 < 50%、过期文档 > 0、文档覆盖 < 100%） → 建议列表
2. 建议格式化 → 对可量化建议计算预期效果（如清理后命中率） → 输出 `--- 建议 ---` 或 `✅ 所有指标健康`
3. review 排序 → 按生命阶段分组（⚠ 濒危: 符合 decay 条件 → 💤 沉默: hits=0 + created > 7d 或 last_hit > 30d → 🆕 新建: created < 7d + hits=0 → ✅ 活跃: last_hit < 30d），同阶段内按 age 降序 → 优先级排序的 review 列表

## 3. 关键决策

| 决策 | 选择 | 为什么 |
|------|------|--------|
| 建议位置 | metrics 输出末尾 | 独立 optimize 命令需额外入口和用户记忆成本，嵌入 metrics 末尾看完数据立刻看到行动 |
| 阈值管理 | 硬编码 | 配置文件增加复杂度且 PRD 明确排除自定义阈值，硬编码保持简单 |
| 预期效果计算 | 只对清理类建议计算 | 规则调整类建议（如检查 scope 推断）效果无法量化，强行估算会产生误导数据 |
