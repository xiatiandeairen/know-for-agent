# /know learn

## 1. 问题

AI agent 在跨会话工作时重复犯同样的错误。上一次会话发现"必须用 Combine 而非 AsyncStream"，下次进来又用 AsyncStream。隐性知识（为什么选 A 不选 B、已知的坑、业务规则的特殊之处）无法从代码和 git 历史直接推导，必须有人踩过才知道。

影响范围：所有需要 agent 跨会话保持认知连续性的项目。learn 是知识的唯一入口，没有它则无知识可用。

## 2. 目标用户

使用 Claude Code 的开发者，在对话中产生了代码和 git 无法表达的高价值知识。

| 场景 | 痛点 |
|------|------|
| 调试发现隐蔽的坑 | 下次 agent 还会踩 |
| 做了技术选型 | 原因没记录，未来被推翻 |
| 纠正 agent 的错误做法 | 下次还得纠正 |
| 解释复杂业务逻辑 | 新会话又要从头解释 |

当前替代方案：

| 方案 | 不足 |
|------|------|
| CLAUDE.md 手写规则 | 手动维护、全量加载、无分类无衰减 |
| Claude 自带 memory | 无结构、无冲突检测、无衰减 |
| 代码注释 | 只能写在代码存在的地方，覆盖不了设计决策和坑 |

## 3. 核心假设

**提供结构化的知识沉淀入口 → 用户愿意花 10 秒确认，换取未来会话不再重复纠正。**

验证方式：用户在真实项目中使用 learn 后，观察是否主动持续使用（而非用一次就放弃）。

## 4. 方案

`/know learn` 一条命令触发，或 AI 检测到隐式信号后主动提议。用户确认后将知识持久化为结构化条目。

用户可感知的变化：
- 对话中说出的隐性知识不再随会话消失
- 知识按标签分类（rationale / constraint / pitfall / concept / reference），按重要程度分层（critical 附详情、memo 仅摘要），自动衰减保持新鲜
- 重复或矛盾的知识会被检测，用户决定更新/保留/合并/跳过
- AI 检测到隐式信号时主动提议，用户确认后才写入——不打断工作流，不自动写入

## 5. 验收标准

- 用户调用 `/know learn`，从对话提取知识、展示完整条目，确认后持久化
- AI 检测到隐式信号时，批量提议，用户选择后逐条处理
- 低价值信息（代码可推导、属于 CLAUDE.md、个人偏好、无结论、一次性）被拦截并展示原因
- 与已有条目冲突时，展示选项（更新/保留/合并/跳过）供用户决策
- 所有写入前必须展示完整条目并获得用户确认

## 6. 排除项

- 无信号或无用户确认不写入（不自动学习）
- 跨项目知识同步
- 知识图谱 / 关联推理
- 批量导入
