# /know learn

<!-- 核心问题: 需求到哪了、验收标准是什么？ -->

## 1. 问题

AI agent 跨会话重复犯错——隐性知识（为什么选 A 不选 B、已知的坑、业务规则特殊之处）无法从代码和 git 推导，会话结束即丢失。影响所有需要 agent 保持认知连续性的项目。learn 是知识唯一入口，没有它则无知识可用。

## 2. 目标用户

使用 Claude Code 的开发者，在对话中产生了代码和 git 无法表达的高价值知识。

| 场景 | 痛点 |
|------|------|
| 调试发现隐蔽的坑 | 下次 agent 还会踩 |
| 做了技术选型 | 原因没记录，未来被推翻 |
| 纠正 agent 的错误做法 | 下次还得纠正 |
| 解释复杂业务逻辑 | 新会话又要从头解释 |

当前替代方案：CLAUDE.md（手动维护、全量加载）、Claude memory（无结构、无衰减）、代码注释（覆盖不了设计决策）。

## 3. 核心假设

**提供结构化的知识沉淀入口 → 用户愿意花 10 秒确认，换取未来会话不再重复纠正。**

验证方式：用户在真实项目中使用 learn 后，观察是否主动持续使用（而非用一次就放弃）。

## 4. 方案

- **Before**: 隐性知识散落在对话中，会话结束即丢失，下次 agent 从零开始 → **After**: `/know learn` 或隐式信号触发，用户确认后知识持久化为结构化条目，下次会话自动可用
- **Before**: 重复知识无检测，多次记录同一件事 → **After**: 冲突自动检测，用户决定更新/保留/合并/跳过
- **Before**: 知识越积越多，加载成本不可控 → **After**: 分层（critical/memo）+ 自动衰减，保持知识库精简

## 5. 验收标准

- 用户调用 `/know learn` → 从对话提取知识、展示完整条目 → 确认后持久化
- AI 检测到隐式信号 → 批量提议 → 用户选择后逐条处理
- 低价值信息被拦截 → 展示拦截原因
- 与已有条目冲突 → 展示选项（更新/保留/合并/跳过）
- 所有写入前 → 展示完整条目 → 用户确认后才写入

## 6. 排除项

- 无信号或无用户确认不写入（不自动学习）
- 跨项目知识同步
- 知识图谱 / 关联推理
- 批量导入
