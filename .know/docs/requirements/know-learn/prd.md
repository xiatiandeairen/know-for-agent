# /know learn

## 问题

AI agent 在跨会话工作时重复犯同样的错误。上一次会话发现"必须用 Combine 而非 AsyncStream"，下次进来又用 AsyncStream。隐性知识（为什么选 A 不选 B、已知的坑、业务规则的特殊之处）无法从代码和 git 历史直接推导，必须有人踩过才知道。

影响范围：所有需要 agent 跨会话保持认知连续性的项目。learn 是知识的唯一入口，没有它则无知识可用。

## 目标用户

使用 Claude Code 的开发者，在对话中产生了代码和 git 无法表达的高价值知识。

使用场景：
- 调试发现一个隐蔽的坑，想让未来的 agent 不再踩
- 做了一个技术选型，想记录原因防止未来被推翻
- 用户纠正 agent 的错误做法，想让纠正永久生效
- 解释了一段复杂的业务逻辑，想让 agent 记住

当前替代方案：CLAUDE.md 手写规则（手动维护、全量加载）、Claude 自带 memory（无结构、无冲突检测、无衰减）、代码注释（只能写在代码存在的地方）。

## 方案

`/know learn` 一条命令触发，或 AI 检测到隐式信号后主动提议。8 步管线：信号检测 → 知识提取 → 路由拦截（过滤代码可推导/属于 CLAUDE.md/个人偏好/无结论/一次性信息）→ 2 问层级评估（重要/备忘）→ 条目生成（tag + scope + summary + 详情文件）→ 冲突检测（关键词预筛 + LLM 语义判断）→ 展示确认 → 写入 `.knowledge/`。

用户可感知的变化：
- 对话中说出的隐性知识不再随会话消失
- 知识按 5 种标签分类（rationale/constraint/pitfall/concept/reference），按重要程度分 2 层（重要附详情文件、备忘仅摘要），自动衰减保持新鲜
- 重复或矛盾的知识会被检测，用户决定更新/保留/合并/跳过
- AI 检测到 6 种隐式信号（用户纠正、技术选型、根因发现、业务逻辑、约束声明、外部集成）时主动提议，用户确认后才写入

## 验收标准

- 显式调用 `/know learn` 后，从对话提取知识、生成条目、确认后持久化到 `.knowledge/`
- AI 检测到 6 种隐式信号时，批量提议（`> [suggest-learn]`），用户选择后逐条处理
- 5 条路由拦截规则正确过滤低价值信息，展示拦截原因（`> [skipped]`）
- 与已有条目冲突时，展示冲突选项（更新/保留/合并/跳过）供用户决策（`> [conflict]`）
- 重要（tier 1）知识附带详情文件（≤220 tokens），备忘（tier 2）仅保留摘要（≤80 chars）
- 所有写入前必须展示完整条目（`> [learn]`）并获得用户确认

## 排除项

- 无信号或无用户确认不写入（不自动学习）
- 跨项目知识同步
- 知识图谱 / 关联推理
- 批量导入

## 协作要求

AI agent 实现信号检测和条目生成，用户负责最终确认。知识的准确性依赖人类判断，AI 不擅自做持久化决策。
