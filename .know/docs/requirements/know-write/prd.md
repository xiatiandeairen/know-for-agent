# /know write

<!-- 核心问题: 需求到哪了、验收标准是什么？ -->

## 1. 问题

和 AI 讨论产品设计、技术方案后，结论散落在对话中，会话结束即丢失。`/know learn` 只写知识条目（≤220 tokens），不适合完整设计文档。影响所有需要将讨论结果沉淀为可追溯文档的场景。

## 2. 目标用户

使用 Claude Code 的开发者，在对话中讨论完产品设计或技术方案后，需要将结果沉淀为结构化文档。

| 场景 | 痛点 |
|------|------|
| 讨论完产品需求 | 结论在对话里，找不到也无法分享 |
| 设计完技术方案 | 没有文档，下次会话又要从头讨论 |
| 做了架构决策 | 决策原因没记录，后续被质疑时无据可查 |

当前替代方案：手动整理（耗时、遗漏）、Notion/Docs AI（脱离代码仓库）、`/know learn`（不适合完整文档）。

## 3. 核心假设

**在 AI 对话上下文中直接生成结构化文档 → 用户愿意用它替代手动整理，因为省力且结果更完整。**

验证方式：用户在讨论完需求/方案后自然地调用 `/know write`，而非回到手动整理。

## 4. 方案

- **Before**: 对话讨论结果散落在聊天记录中，会话结束即丢失 → **After**: `/know write` 一条命令，AI 从对话推断类型并按模板生成完整文档
- **Before**: 手动整理文档耗时且格式不统一 → **After**: 文档自动结构化，缺失内容标 TBD，与代码同 git 仓库
- **Before**: 已有文档和新讨论结果之间无关联 → **After**: CLAUDE.md 索引自动维护，父子文档级联标记更新

## 5. 验收标准

- 用户调用 `/know write` → AI 正确推断文档类型和参数 → 确认后生成文档
- 生成文档覆盖模板所有章节 → 内容来自对话而非编造
- 版本级文档支持递增（v1 → v2） → 需求级文档原地覆写
- CLAUDE.md 索引正确更新 → 包含路径、日期、层级关系
- 所有写入前 → 展示完整预览 → 用户确认后才写入
- 已有文档更新 → 只修改对话涉及章节 → 未讨论章节不变
- 父文档写入后 → 直接子文档标记 `⚠ needs update` → 子文档更新后自动清除

## 6. 排除项

- 递归级联（只标记直接子文档，不穿透）
- 文档联动自动重写（只标记，不自动生成新内容）
- 自定义模板（Later）
