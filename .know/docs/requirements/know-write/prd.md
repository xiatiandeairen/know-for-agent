# /know write

## 1. 问题

和 AI 讨论产品设计、技术方案后，结论散落在对话中，会话结束即丢失。`/know learn` 只写知识条目，不适合完整设计文档。

影响范围：所有需要将对话讨论结果沉淀为可追溯文档的场景。没有 write，讨论结果只能手动整理或丢失。

## 2. 目标用户

使用 Claude Code 的开发者，在对话中讨论完产品设计或技术方案后，需要将结果沉淀为结构化文档。

| 场景 | 痛点 |
|------|------|
| 讨论完产品需求 | 结论在对话里，找不到也无法分享 |
| 设计完技术方案 | 没有文档，下次会话又要从头讨论 |
| 做了架构决策 | 决策原因没记录，后续被质疑时无据可查 |

当前替代方案：

| 方案 | 不足 |
|------|------|
| 手动整理 | 耗时，容易遗漏，格式不统一 |
| Notion AI / Docs AI | 脱离开发上下文，不和代码仓库联动 |
| `/know learn` | ≤220 tokens，不适合完整文档 |

## 3. 核心假设

**在 AI 对话上下文中直接生成结构化文档 → 用户愿意用它替代手动整理，因为省力且结果更完整。**

验证方式：用户在讨论完需求/方案后自然地调用 `/know write`，而非回到手动整理。

## 4. 方案

`/know write` 一条命令触发，AI 从对话推断文档类型和参数，按模板生成结构完整的文档，写入 `.know/docs/`，CLAUDE.md 自动维护索引。

用户可感知的变化：
- 对话讨论结果不再随会话消失，一条命令生成完整文档
- 文档按模板结构化，覆盖关键章节，缺失内容标"TBD"
- 文档与代码同处一个 git 仓库，有版本追溯
- CLAUDE.md 索引自动维护，agent 下次会话能发现已有文档
- 重复或矛盾的知识会被检测，用户决定更新/保留/合并/跳过

## 5. 验收标准

- 用户调用 `/know write`，AI 正确推断文档类型和参数，确认后生成文档
- 生成的文档覆盖模板所有章节，内容来自对话而非编造
- 项目版本级文档支持版本递增（v1 → v2），需求级文档原地覆写
- CLAUDE.md 索引正确更新，包含路径、日期、层级关系
- 所有写入前展示完整文档预览，用户确认后才写入
- 已有文档更新时，只修改对话涉及的章节，未讨论章节保持不变
- 更新写入后自动追加 changelog 条目（日期 + 变更摘要）
- 父文档写入后，CLAUDE.md 索引中直接子文档自动标记 `⚠ needs update`
- 子文档更新后自动清除 `⚠ needs update` 标记

## 6. 排除项

- 递归级联（只标记直接子文档，不穿透）
- 文档联动自动重写（只标记，不自动生成新内容）
- 自定义模板（Later）
