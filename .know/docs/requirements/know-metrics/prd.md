# know metrics



## 1. 问题

know 的三条管线（learn/recall/write）已跑通，但没有质量度量。用户不知道存的知识是否有用、recall 是否在防错、文档体系是否跟上进度。每个使用 know 的开发者每次会话都面临这个盲区。v1 M3/M4 验证在即，没有数据无法判断。

## 2. 目标用户

使用 know skill 的开发者，想了解知识体系的健康状况。


| 场景          | 痛点              |
| ----------- | --------------- |
| 存了 20 条知识   | 不知道哪些被用上了、哪些是噪音 |
| recall 每天运行 | 不知道是否真的帮自己避了错   |
| roadmap 更新了 | 不知道下游 PRD 是否过期  |


当前替代方案：手动 `know-ctl stats` 只看条目数和分布，无使用质量数据。

## 3. 核心假设

**提供管线运行质量指标 → 用户能判断"哪里需要优化"并采取行动（清理低效知识、调整 scope 规则、更新过期文档）。**

验证方式：用户看到 metrics 输出后，做出 1 次有依据的优化决策（如删除命中率为 0 的条目）。

## 4. 方案

- **Before**: 只能看条目数，不知道知识是否被用上 → **After**: `know-ctl metrics` 展示命中率，一眼看出哪些知识是噪音
- **Before**: 不知道 recall 是否在工作 → **After**: metrics 展示防御次数和覆盖率，验证 recall 价值
- **Before**: 不知道文档体系是否过期 → **After**: metrics 展示过期文档数，提醒需要更新的文档

### 指标定义


| 管线     | 核心问题    | 指标    | 计算方式                            |
| ------ | ------- | ----- | ------------------------------- |
| learn  | 存的有用吗？  | 命中率   | hits > 0 的条目数 / 总条目数            |
| learn  | 存的有用吗？  | 衰减率   | 被 decay 删除或降级的数量 / 总创建数         |
| recall | 帮我避错了吗？ | 防御次数  | tm=active:defensive 条目的 hits 总和 |
| recall | 帮我避错了吗？ | 覆盖率   | 被查询过的 scope 数 / 总 scope 数       |
| write  | 文档跟上了吗？ | 过期文档数 | CLAUDE.md 中 ⚠ needs update 标记数  |
| write  | 文档跟上了吗？ | 文档覆盖率 | 有文档的里程碑 / 总里程碑数                 |


### 数据来源


| 指标    | 数据源                         | 现有/新增 |
| ----- | --------------------------- | ----- |
| 命中率   | index.jsonl hits 字段         | 现有    |
| 衰减率   | 需要记录总创建数                    | 新增    |
| 防御次数  | index.jsonl hits + tm 字段    | 现有    |
| 覆盖率   | 需要记录查询过的 scope              | 新增    |
| 过期文档数 | CLAUDE.md ⚠ 标记              | 现有    |
| 文档覆盖率 | roadmap 里程碑表 + CLAUDE.md 索引 | 现有    |

## 5. 验收标准

- 用户运行 `know-ctl metrics` → 输出 learn/recall/write 三组指标
- learn 组展示命中率和衰减率 → 数字来自 index.jsonl 真实数据
- recall 组展示防御次数和覆盖率 → 防御次数来自 active:defensive 条目的 hits
- write 组展示过期文档数和文档覆盖率 → 数据来自 CLAUDE.md 和 roadmap
- 无条目时 → 各指标显示 0，不报错
- 有条目无命中时 → 命中率显示 0%，不隐藏

## 6. 排除项

- 不支持历史趋势（只展示当前快照）
- 不支持跨项目聚合
- 不支持自定义指标
- 仪表盘 UI 推迟到 v3
