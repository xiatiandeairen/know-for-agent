# know 产品路线图

<!-- 核心问题: 产品到哪了、下一步往哪走？ -->

## 1. 产品愿景

| | |
|---|---|
| **解决什么问题** | AI agent 跨会话丢失隐性知识和设计讨论结果，重复犯错 |
| **给谁用** | 使用 Claude Code 等 AI 编程工具的开发者 |
| **核心差异** | 知识与代码同源同版本、分层控制 token 成本、自动衰减保持新鲜 |

## 2. 版本规划

### v3 — 可靠性

| 维度 | 标准 |
|------|------|
| **功能** | know-ctl 有自测、文档有一致性检查、write 流程规则完整无遗漏 |
| **质量** | self-test 全部通过、check 输出 0 个误报 |
| **用户** | 日常使用中不再遇到路径错误、版本判定错误、文档不同步 |

#### 里程碑

| # | 里程碑 | 验证点 | 进度 | 需求 |
|---|--------|--------|------|------|
| M1 | **能测** — know-ctl 自动化验证 | `know-ctl self-test` 覆盖核心命令（append/query/hit/decay/metrics/history），全部通过 | 1/1 | [selftest](../requirements/know-selftest/prd.md) |
| M2 | **能查** — 文档一致性检查 | `know-ctl check` 检测模版与项目文档的结构偏差，输出具体不一致项 | 1/1 | [check](../requirements/know-check/prd.md) |
| M3 | **能守** — 流程规则完整性 | write 流程的关键判定点（版本、类型、父子关系）有明确规则且无遗漏 | 1/1 | [flowrules](../requirements/know-flowrules/prd.md) |

## 3. 当前版本

### 包含

| 能力 | 解决什么问题 |
|------|-------------|
| **自测** | know-ctl 改动后无验证手段 → self-test 命令一键验证核心功能 |
| **一致性检查** | 模版改了但项目文档没跟上 → check 命令检测结构偏差 |
| **流程规则** | write 判定规则不完整导致写错文件 → 关键判定点有明确规则 |

### 排除

- 自动修复不一致（只检测，不自动改）
- 跨项目一致性检查
- CI/CD 集成

## 4. 风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| 自测覆盖范围难定义 | 测太少没用，测太多维护负担 | 只覆盖核心命令的 happy path + 关键边界 |
| 一致性检查误报 | 用户忽略 check 输出 | 只检查结构（section 数量/名称），不检查内容 |
| 流程规则难穷尽 | 新场景仍会遗漏 | 每次发现遗漏立即补规则，形成积累 |
