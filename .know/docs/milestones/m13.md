# M13 — recall 价值量化

## 1. 目标

### 完成定义

recall 的防错价值可通过数据证明：防御次数 >10，且至少有 3 次可追溯的"recall 命中 → 避免了具体错误"案例

### 价值目标

用户可以用数据回答"know 到底有没有用"——不是感觉有用，是数据证明有用

### 关键假设

- **假设**: 在知识库质量提升（M11）+ 匹配优化（M12）后，recall 可以在日常使用中持续产生防错价值
- **类型**: 商业价值
- **验证方式**: 连续使用 2 周，统计 recall 防御次数和具体防错案例，判断 ROI 是否正向

## 2. 计划

### 预期交付物

- 数据: 2 周使用期间的 recall 防御记录（events.jsonl）
- 文档: ≥3 个具体防错案例（知识条目 + 触发场景 + 避免了什么错误）
- 数据: 最终 metrics 快照（命中率/覆盖率/防御次数）

### 达标标准

| 维度 | 标准 |
|------|------|
| 功能完整性 | 防御次数 >10（know-ctl metrics 实测） |
| 质量 | ≥3 个具体防错案例可追溯（events.jsonl + 知识条目对照） |
| 性能与稳定性 | 连续 2 周使用无 recall 相关阻塞或异常 |
| 安全 | 本里程碑不涉及 |
| 可维护性 | 本里程碑不涉及 |

### 预估投入

2 周持续使用观察（非全职开发，日常使用中积累数据）

### 决策路径

- **通过**: 防御 >10 + ≥3 案例 → v4 完成，评估是否进入 v5（方向：多项目支持/社区推广/更多管线）
- **未通过**: 防御 <10 → 分析原因：知识不够多？使用场景不够频繁？recall 触发条件太严？针对性调整

### 风险

依赖: M11+M12 已完成且指标达标；风险：2 周使用期间项目开发活跃度不足，导致 recall 触发次数不够
