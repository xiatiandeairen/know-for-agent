# M10 — write 流程规则完整性

## 1. 目标

### 完成定义

write 流程中所有关键判定点（版本判定、类型推断、父子关系、create/update 模式切换）有明确规则且无遗漏

### 价值目标

用户执行 write 流程时，每个判定点行为可预测，不出现未定义的歧义分支

### 关键假设

- **假设**: 枚举所有判定点并显式定义规则可以消除 write 的不确定行为
- **类型**: 技术可行性
- **验证方式**: 用边界 case（无父文档的 tech、已存在文件的 create、多类型推断）测试 write 流程，检查是否全部命中明确规则

## 2. 计划

### 预期交付物

- 文档: write.md 中完善的判定规则集（版本/类型/父子/模式切换）
- 文档: 模版文件中的结构化质量约束

### 达标标准

| 维度 | 标准 |
|------|------|
| 功能完整性 | 4 个关键判定点（版本/类型/父子/模式切换）均有明确规则且无遗漏 |
| 质量 | 规则无矛盾、无歧义，边界 case 全部命中明确规则 |
| 性能与稳定性 | 本里程碑不涉及 |
| 安全 | 本里程碑不涉及 |
| 可维护性 | 新增文档类型时规则扩展路径明确（路径表 + 模版 + write.md 三处同步） |

### 预估投入

1 天全职开发

### 决策路径

- **通过**: v3 可靠性完成，评估是否进入 v4（方向待定：推广/多项目支持/更多管线）
- **未通过**: 补全遗漏规则，确保 write 流程行为完全可预测

### 风险

依赖对 write 流程所有分支路径的完整梳理；风险：模版内容质量约束可能比流程规则更复杂，范围可能膨胀
