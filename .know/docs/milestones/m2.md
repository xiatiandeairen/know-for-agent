# M2 — 验证 write 管线端到端可行性

## 1. 目标

### 完成定义

用户在对话中讨论需求/架构/方案后，执行 `/know write` 可生成结构完整的文档并写入正确路径。

### 价值目标

用户可通过一条命令将对话中的决策和方案沉淀为结构化文档，无需手动整理。

### 关键假设

- **假设**: AI 可以从非结构化对话中提取足够信息填充结构化文档模版
- **类型**: 技术可行性
- **验证方式**: 在 know 项目的需求讨论对话中执行 write，检查生成文档的结构完整性和内容准确性

## 2. 计划

### 预期交付物

- 代码: write.md workflow 文件（6 步管线定义）
- 文档: 10 种文档类型模版（roadmap/prd/tech/arch/ui/schema/decision/ops/capabilities/marketing）
- 数据: `.know/docs/` 目录结构 + 父子关系索引

### 达标标准

| 维度 | 标准 |
|------|------|
| 功能完整性 | 对话→`/know write`→完整文档 + 路径正确，10 种文档类型全部可用 |
| 质量 | 文档结构完整，内容来自对话而非捏造 |
| 性能与稳定性 | 单次 write 操作 <60s |
| 安全 | 不覆盖用户未确认的文档内容 |
| 可维护性 | 新增文档类型只需加模版文件 + 路径表一行 |

### 预估投入

1 天全职开发

### 决策路径

- **通过**: 进入 M3 验证 recall 防错效果，learn + write 两条管线已就绪
- **未通过**: 排查 write 管线阻塞点，暂不验证 recall

### 风险

依赖: AI 对对话内容的理解能力；风险: 模版过于宽松可能导致内容空洞
