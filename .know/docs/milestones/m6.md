# M6 — 知识全生命周期可追踪

## 1. 目标

### 完成定义

用户可查询任意一条知识从存入→命中→衰减→删除的完整事件链。

### 价值目标

用户可按 slug 追踪每条知识的完整生命周期，理解知识的使用情况和演变过程。

### 关键假设

- **假设**: append-only 事件日志可以完整重建知识生命周期
- **类型**: 技术可行性
- **验证方式**: 选取一条经历过 hit 和 decay 的知识，用 history 命令查询，检查事件链完整性

## 2. 计划

### 预期交付物

- 代码: know-ctl history 命令
- 数据: events.jsonl 中的完整事件链记录

### 达标标准

| 维度 | 标准 |
|------|------|
| 功能完整性 | history 命令按 slug 查询，输出完整事件链轨迹 |
| 质量 | 事件链完整无断裂（events.jsonl 引入后创建的条目） |
| 性能与稳定性 | history 查询 <3s |
| 安全 | 本里程碑不涉及 |
| 可维护性 | 事件格式可扩展，新增事件类型无破坏性 |

### 预估投入

1 天全职开发

### 决策路径

- **通过**: 进入 M7 优化闭环验证，生命周期数据为优化决策提供依据
- **未通过**: 补全事件记录机制后重新验证

### 风险

依赖 events.jsonl 事件记录完整性；风险：早期条目（events.jsonl 引入前创建的）缺少 created 事件，补全成本高
