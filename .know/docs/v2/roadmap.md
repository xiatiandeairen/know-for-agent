# know 产品路线图

<!-- 核心问题: 产品到哪了、下一步往哪走？ -->

## 1. 产品愿景

| | |
|---|---|
| **解决什么问题** | AI agent 跨会话丢失隐性知识和设计讨论结果，重复犯错 |
| **给谁用** | 使用 Claude Code 等 AI 编程工具的开发者 |
| **核心差异** | 知识与代码同源同版本、分层控制 token 成本、自动衰减保持新鲜 |

## 2. 版本规划

### v2 — 可观测性

| 维度 | 标准 |
|------|------|
| **功能** | learn/write/recall 各有核心指标，可通过命令查看 |
| **质量** | 指标数据来自真实使用，非模拟数据 |
| **用户** | 看到指标后能判断"哪里需要优化"并采取行动 |

#### 里程碑

| # | 里程碑 | 验证点 | 进度 | 需求 |
|---|--------|--------|------|------|
| M1 | **能量** — 核心指标定义 + 数据收集 | `know-ctl metrics` 输出 learn/recall/write 各 2-3 个指标，数据来自真实使用 | 1/1 | [metrics](../requirements/know-metrics/prd.md) |
| M2 | **能看** — 知识生命周期可追踪 | 一条知识从存入 → 命中 → 衰减/删除的完整轨迹可查询 | 1/1 | [lifecycle](../requirements/know-lifecycle/prd.md) |
| M3 | **能调** — 基于数据做出一次优化决策 | 看到指标后发现问题，调整规则，指标改善 | 1/1 | [optimize](../requirements/know-optimize/prd.md) |

## 3. 当前版本

### 包含

| 能力 | 解决什么问题 |
|------|-------------|
| **度量体系** | 不知道管线质量好不好 → 核心指标可查，数据驱动优化 |
| **反馈闭环** | 知识存了不知道有没有用 → 生命周期可追踪 |

### 排除

- 自动优化（只提供数据，决策由用户做）
- 跨项目指标聚合
- 仪表盘 UI

## 4. 风险与依赖

| 风险 | 影响 | 缓解 |
|------|------|------|
| v1 使用数据不足 | 指标无意义，缺少基线 | v1 M4 先完成（连续使用 ≥1 周） |
| 度量开销 | 每次操作额外 IO | 复用已有 hits/revs 字段，增量记录 |
