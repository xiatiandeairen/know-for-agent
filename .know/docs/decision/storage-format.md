# 存储格式选型 决策记录

## 1. 背景

**触发事件:** know 项目初始设计阶段，需要确定知识条目的持久化存储方案。知识库需要在 Claude Code agent 会话间持久保存结构化条目，并支持增删改查操作。

**约束:**
- 零外部依赖：作为 Claude Code skill 运行，不能依赖数据库服务或编译型工具
- Git 可追踪：所有数据变更需要在 git diff 中可读、可 review
- 并发安全：多个 agent 会话可能同时操作知识库
- 纯 bash 实现：CLI 工具用 shell 脚本实现，存储格式必须用标准 Unix 工具可操作

**决策范围:** 选择 index 数据（结构化条目元信息）的存储格式，不涉及详情文件（已确定为独立 .md 文件）。

## 2. 决策

**我们决定:** 采用 JSONL（每行一条 JSON）作为 index 存储格式，配合独立的 .md 文件存放详情内容。

JSONL 是唯一同时满足零依赖、git 友好、行级追加无锁冲突三个硬约束的方案。在当前规模（百级条目）下，线性扫描的性能代价可忽略。

## 3. 备选方案

### 方案 A: JSONL（每行一条 JSON）

**优点:**
- 零依赖，jq + bash 即可完成所有操作
- Git 友好，每条变更在 diff 中独立可读，合并冲突易解决
- 追加写入天然无锁，多进程并发 append 安全

**缺点:**
- 不支持复杂查询（join、聚合），只能线性扫描 + jq filter
- 更新和删除需要重写整个文件（读→改→写→rename）
- 无 schema 强制校验，数据一致性靠应用层保证

### 方案 B: SQLite

**优点:**
- 支持复杂查询、索引、事务，查询性能随数据量线性扩展
- 内置 schema 校验和数据完整性约束

**缺点:**
- 二进制文件，git diff 不可读，合并冲突无法手动解决
- 需要 sqlite3 CLI 依赖，部分环境可能未安装
- 文件级锁定，并发写入需要额外处理

### 方案 C: 单个 JSON 文件（数组）

**优点:**
- 结构直观，整个数据集一目了然
- 标准 JSON，任何语言和工具原生支持

**缺点:**
- 追加需要读→解析→修改→序列化→写回，不支持行级追加
- Git diff 中一次修改可能影响多行（数组重排、逗号变化）
- 并发写入必然冲突，大文件时解析开销随条目数增长

## 4. 影响

**正面影响:**
- 零依赖部署：任何安装了 jq 的系统均可运行，无需额外安装步骤
- Git 可追踪：知识库变更历史完全透明，支持 code review 和版本回溯
- 并发安全追加：多个 agent 会话可同时 append 条目，无数据丢失风险

**负面影响:**
- 无复杂查询能力：不支持多字段联合索引、聚合统计，查询效率为 O(n)
- 更新/删除操作需要全文件重写：delete、update、decay 操作涉及临时文件 + rename

**后续行动:**
- 如果条目数超过 1000，评估是否需要引入分片或缓存机制
- 应用层维护字段校验逻辑（append 时校验 5 个必填字段）
